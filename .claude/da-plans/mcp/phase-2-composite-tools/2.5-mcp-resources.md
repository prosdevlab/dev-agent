# Part 2.5: MCP Resources — Ambient Codebase Context

See [overview.md](overview.md) for architecture context.

## Goal

Add MCP resources so that AI assistants start every conversation with ambient
codebase awareness — no tool calls needed for basic structural knowledge.

## The problem

Today, every conversation starts cold. The LLM doesn't know what packages exist,
what the hot paths are, or what conventions the codebase follows. It must call
`dev_map` and `dev_patterns` first, burning tool calls and tokens on orientation
before it can do real work.

MCP resources solve this. They're passive context the client loads automatically.
The LLM reads them like it reads system instructions — always available, zero
round-trips.

### `packages/mcp-server/src/resources/conventions-resource.ts` (new)

~80-100 lines. Extracts dominant coding patterns from the pattern analysis

```typescript
export const CONVENTIONS_RESOURCE_URI = 'dev-agent://conventions';

export function getConventionsResourceDefinition(): ResourceDefinition {
  return {
    uri: CONVENTIONS_RESOURCE_URI,
    name: 'Codebase Conventions',
    description:
      'Coding patterns and conventions: error handling, imports, naming, ' +

    mimeType: 'text/markdown',

  indexer: RepositoryIndexer,
  searchService: SearchService

    uri: CONVENTIONS_RESOURCE_URI,
    mimeType: 'text/markdown',

}
```

### Convention extraction approach

The existing `PatternAnalysisService.analyzeFilePatterns()` compares one file
against similar files. There is no existing function that aggregates dominant
conventions across the entire codebase. We need a new function:

```typescript
// In packages/core/src/services/pattern-analysis-service.ts
export async function extractDominantConventions(
  searchService: SearchService,
  options?: { sampleSize?: number }
): Promise<ConventionSummary> { ... }
```

**Algorithm:**
1. Sample files by PageRank (top N hot-path files — they define conventions
   by being the most-referenced). Default `sampleSize`: 15 files.
2. For each sampled file, call the existing `analyzeFilePatterns()` to get
   its error handling, import, and naming patterns.
3. Aggregate: for each pattern dimension (error handling, imports, naming,
   types), tally approaches across sampled files. The majority approach
   (>50% of samples) becomes the "convention."
4. Format as concise markdown.

**Why sample by PageRank, not random:** Hot-path files are the ones LLMs are
most likely to modify or reference. Their patterns define "what the codebase
does" more accurately than random files.

**Risk adjustment:** This is a new aggregation algorithm, not a thin wrapper.
Adjust risk from "Low" to **Low-Medium**. The individual pattern analysis
exists; the aggregation is straightforward tallying.

Key constraints:
- Generated from pattern analysis, not hardcoded
- Samples top 15 files by PageRank for convention extraction
- Capped at ~500 tokens (measured via `estimateTokensForText` from
  `packages/mcp-server/src/formatters/utils.ts`)
- Falls back to "No conventions extracted yet — run `dev index` first"

### `packages/mcp-server/src/server/mcp-server.ts` (modify)

**Dependency injection for resources.** `MCPServer` currently receives adapters
(which hold their own deps) but has no direct access to `indexer` or
`searchService`. Resources need these.

Approach: add a `ResourceProvider` to `MCPServerConfig`. This parallels how
`adapters` are passed — the entry point constructs the provider, the server
calls it. No coupling between server internals and core services.

```typescript
// In MCPServerConfig:
export interface ResourceProvider {
  listResources(): ResourceDefinition[];
  readResource(uri: string): Promise<ResourceContent>;
}

export interface MCPServerConfig {
  // ... existing fields
  resourceProvider?: ResourceProvider;
}
```

The entry point (`bin/dev-agent-mcp.ts`) constructs the provider with the
services it already has:

```typescript
// In dev-agent-mcp.ts, after creating adapters:
import { createResourceProvider } from '../src/resources/index.js';

const resourceProvider = createResourceProvider({
  indexer,
  searchService,
  graphPath: filePaths.dependencyGraph,
});

const server = new MCPServer({
  // ... existing config
  resourceProvider,
});
```

The `createResourceProvider` factory lives in `resources/index.ts`:

```typescript
export function createResourceProvider(deps: {
  indexer: RepositoryIndexer;
  searchService: SearchService;
  graphPath?: string;
}): ResourceProvider {
  return {
    listResources: () => [
      getMapResourceDefinition(),
      getConventionsResourceDefinition(),
    ],
    readResource: async (uri: string) => {
      switch (uri) {
        case MAP_RESOURCE_URI:
          return readMapResource(deps.indexer, deps.graphPath);
        case CONVENTIONS_RESOURCE_URI:
          return readConventionsResource(deps.indexer, deps.searchService);
        default:
          throw new Error(`Unknown resource: ${uri}`);
      }
    },
  };
}
```

Server wiring:

```typescript
// In handleInitialize:
resources: { supported: !!this.resourceProvider },

// In handleResourcesList:
private handleResourcesList() {
  if (!this.resourceProvider) return { resources: [] };
  return { resources: this.resourceProvider.listResources() };
}

// In handleResourcesRead (replace the throw):
private async handleResourcesRead(params: { uri: string }) {
  if (!this.resourceProvider) {
    throw createError(ErrorCode.ResourceNotFound, `Resources not available`);
  }
  try {
    const content = await this.resourceProvider.readResource(params.uri);
    return { contents: [content] };
  } catch {
    throw createError(ErrorCode.ResourceNotFound, `Unknown resource: ${params.uri}`);
  }
}
```

This keeps `MCPServer` decoupled from core — it only knows the
`ResourceProvider` interface, not `RepositoryIndexer` or `SearchService`.
```

## Edge cases

- **Index not built yet:** Return minimal resource with message:
  "Index not available. Run `dev index` to enable full codebase context."
- **Antfly down:** Map resource falls back to filesystem-only structure
  (packages + file counts from glob, no hot paths). Conventions resource
  returns "Conventions unavailable — Antfly not running."
- **Very large repo (50k+ files):** Map resource shows top-level packages
  only (depth 1). Hot paths limited to top 5.
- **Resource read fails:** Return error content, never crash the server.

### Transport-level round-trip test

Resources bypass `AdapterRegistry` — they're handled directly in `MCPServer`.
Add a transport-level test that sends raw JSON-RPC through the message handler:

```typescript
// In packages/mcp-server/src/server/__tests__/mcp-server-resources.test.ts
describe('MCP Server — resources round-trip', () => {
  it('resources/list returns resource definitions via JSON-RPC', async () => {
    const server = createTestServer({ resourceProvider: mockProvider });
    const response = await server.handleMessage({
      jsonrpc: '2.0', id: 1, method: 'resources/list', params: {},
    });
    expect(response.result.resources).toHaveLength(2);
    expect(response.result.resources[0]).toHaveProperty('uri');
    expect(response.result.resources[0]).toHaveProperty('mimeType');
  });

  it('resources/read returns content via JSON-RPC', async () => {
    const server = createTestServer({ resourceProvider: mockProvider });
    const response = await server.handleMessage({
      jsonrpc: '2.0', id: 2, method: 'resources/read',
      params: { uri: 'dev-agent://map' },
    });
    expect(response.result.contents).toHaveLength(1);
    expect(response.result.contents[0].text).toContain('Codebase');
    expect(response.result.contents[0].mimeType).toBe('text/markdown');
  });

  it('resources/read with unknown URI returns error', async () => {
    const server = createTestServer({ resourceProvider: mockProvider });
    const response = await server.handleMessage({
      jsonrpc: '2.0', id: 3, method: 'resources/read',
      params: { uri: 'dev-agent://unknown' },
    });
    expect(response.error.code).toBe(-32002); // ResourceNotFound
  });
});
```
